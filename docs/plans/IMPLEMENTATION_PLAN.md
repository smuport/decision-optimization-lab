# 递进式实施计划

> 本计划用于指导《工程系统决策与优化》在线实验平台从课程资料整理到正式运行的分阶段建设。
> 核心原则：先让课程案例可被平台理解，再让平台可被学生使用，最后再做并发、安全、运维和教学分析增强。

---

## 一、总体路线

```text
阶段 0：课程资产盘点与标准化
阶段 1：静态课程门户
阶段 2：本地评测原型
阶段 3：最小在线提交平台
阶段 4：评测沙箱与安全增强
阶段 5：教师教学闭环
阶段 6：正式部署与运维
阶段 7：持续扩展与质量提升
```

推荐先用 3 个代表案例跑通闭环：

| 案例 | 类型 | 选择原因 |
|------|------|----------|
| case_01 生产分配问题 | 精确线性规划 | 可验证目标值、变量、资源约束、影子价格 |
| case_04 分配问题 | 组合优化/匹配 | 可验证分配可行性、总成本、非方阵处理 |
| case_16 模拟退火 TSP | 启发式算法 | 可验证路线合法性、GAP、运行时间、收敛曲线 |

完成这 3 类后，再把 case_02 ~ case_18 逐步接入。

### 当前阶段调整（Week3）

Week2 已用 case01 验证学生提交与教师查看闭环。Week3 暂不扩展 case04、case16 或其他案例，优先完成教学管理控制面：

```text
ADMIN 管理 Case/Exercise
→ TEACHER 管理 ClassSection 的案例可见性
→ TEACHER 将 Exercise 发布为 Assignment
→ STUDENT 按班级读取和提交
```

当前权威关系为 `Case → Exercise`、`ClassSection → SectionCaseRelease → Case` 和 `ClassSection → Assignment → Exercise`。模板、数据集、rubric 和 validator 归属 Exercise。完成该控制面并经过真实班级试用后，再决定案例扩展顺序。

## 二、阶段 0：课程资产盘点与标准化

**目标**：把现有案例整理成平台可识别、可下载、可评测、可展示的标准资产。

**建议工期**：1 周。

**主要任务**

1. 盘点全部案例目录，确认 README、数据集、参考代码、练习文件是否完整。
2. 将案例按评测方式分为：精确建模题、启发式算法题、理解分析题、混合题。
3. 为每个案例建立 `case_manifest.json`。
4. 为首批 3 个案例编写 `template.py`、`validator.py`、`rubric.json`。
5. 统一学生提交接口，优先采用 `solve(data, params) -> dict`。
6. 统一评测结果 JSON 结构。
7. 明确允许依赖清单，例如 `pulp`、`numpy`、`matplotlib`、`scipy`、`networkx`。

**交付物**

```text
course-assets/
├── manifests/course_manifest.json
└── cases/
    ├── case_01/
    ├── case_04/
    └── case_16/
```

每个首批案例至少包含：

```text
case_manifest.json
README.md
template.py
validator.py
rubric.json
datasets/data_small.json
datasets/data_medium.json
datasets/data_large.json
hidden/sample_hidden.json
```

**验收标准**

- 能用统一 manifest 读取案例标题、分类、难度、知识点、数据集和依赖。
- 每个首批案例都有明确输入、输出、评分规则。
- 教师能说明该案例自动评分与人工评分各占多少。
- 隐藏数据和教师参考实现不会暴露给学生端。

**风险与处理**

| 风险 | 处理方式 |
|------|----------|
| 现有 `solve.py` 输出以自然语言为主 | 新增结构化 `template.py` 和 `validator.py`，不直接解析旧 stdout |
| 不同案例评分差异大 | 评分规则放入 `rubric.json`，按案例插件化 |
| 启发式算法没有精确最优 | 使用已知最优、参考基准或教师设定阈值，重点检查可行性和 GAP |

## 三、阶段 1：静态课程门户

**目标**：先让课程资料可在线访问，学生能阅读文档、下载数据和模板。

**建议工期**：1-2 周。

**主要任务**

1. 建立前端基础项目。
2. 实现课程首页、案例列表、案例详情。
3. 支持 Markdown 和数学公式渲染。
4. 支持数据集和模板下载。
5. 从 `course_manifest.json` 生成案例导航。
6. 将现有 README、图片、数据集迁移或映射到前端可访问路径。

**交付物**

- 静态课程门户。
- 案例列表页。
- 案例详情页。
- 数据集/模板下载入口。

**验收标准**

- 学生打开网页即可看到课程案例。
- 至少 3 个首批案例可正常阅读、下载数据和下载模板。
- 数学公式、代码块、表格显示正常。
- 移动端和桌面端基本可用。

**技术建议**

第一阶段可以完全不接后端数据库，直接读取静态 manifest。这样能最快验证内容组织方式。

## 四、阶段 2：本地评测原型

**目标**：先在命令行跑通评测闭环，再搬到在线平台。

**建议工期**：2 周。

**主要任务**

1. 建立 `runner/evaluate.py` 统一入口。
2. 建立标准评测流程：读取 manifest → 复制提交 → 执行 → 调用 validator → 输出 result JSON。
3. 为 case_01 实现线性规划评测器。
4. 为 case_04 实现分配问题评测器。
5. 为 case_16 实现 TSP 评测器。
6. 保存 stdout/stderr、运行时间、内存、结果 JSON。
7. 准备公开数据和隐藏数据的评测模式。

**建议命令**

```bash
python runner/evaluate.py \
  --case case_04 \
  --dataset data_small.json \
  --submission submissions/demo_solution.py
```

**标准结果结构**

```json
{
  "status": "SUCCESS",
  "isFeasible": true,
  "objective": 123.0,
  "optimalObjective": 123.0,
  "gap": 0.0,
  "runtimeMs": 240,
  "memoryMb": 96,
  "score": 95,
  "scoreItems": {
    "feasibility": 20,
    "quality": 40,
    "efficiency": 15,
    "robustness": 10,
    "report": 10
  },
  "metrics": {},
  "visualization": {},
  "messages": []
}
```

**验收标准**

- 3 个代表案例均可通过命令行评测。
- 故意提交错误解时，评测器能给出明确失败原因。
- 隐藏数据不会进入学生提交目录。
- 结果 JSON 可以被前端直接展示。

**风险与处理**

| 风险 | 处理方式 |
|------|----------|
| 学生代码不按接口返回 | 模板中强约束 `solve()`，提交前做接口检查 |
| 评测器和参考解混杂 | validator 只保留验证逻辑，reference_solution 单独存放 |
| stdout 太长影响存储 | 截断展示，完整日志保存为 artifact |

## 五、阶段 3：最小在线提交平台

**目标**：学生能登录、提交代码、查看评测结果；教师能查看班级提交情况。

**建议工期**：3-4 周。

**主要任务**

1. 建立后端 API 项目。
2. 建立数据库 MVP 表：用户、课程、班级、选课、案例、实验、作业、提交、运行结果、成绩、报告。
3. 实现学生名单导入和首次激活。
4. 实现案例和实验任务 API。
5. 实现代码上传和提交记录。
6. 后端调用本地 runner 完成评测。
7. 前端展示提交状态、每个数据集的结果和日志。
8. 教师端展示班级提交列表和通过率。

**MVP 功能边界**

必做：

- 登录。
- 当前课程首页。
- 案例详情。
- 模板下载。
- 代码上传。
- 评测结果。
- 提交历史。
- 教师查看提交。

后置：

- 排行榜。
- WebSocket 实时推送。
- MinIO。
- BullMQ。
- 复杂通知中心。
- 多文件在线工程。

**验收标准**

- 一个学生账号可以完成从登录到提交的全过程。
- 教师可以看到该学生的提交和分数。
- 至少 3 个代表案例可在线评测。
- 评测失败时，学生能看到明确错误：语法错误、接口错误、超时、答案不可行、GAP 过大等。

## 六、阶段 4：评测沙箱与安全增强

**目标**：把“能跑”提升为“可控、安全、可恢复”。

**建议工期**：2-4 周。

**主要任务**

1. 构建课程评测 Docker 镜像。
2. 固定允许依赖版本。
3. 禁止网络访问。
4. 限制 CPU、内存、进程数、运行时间。
5. 使用非 root 用户运行学生代码。
6. 将数据集只读挂载，将输出目录单独挂载。
7. 每次提交使用独立临时目录。
8. 评测结束强制清理容器和临时文件。
9. 评估 rootless Docker、独立评测机或隔离 runner。
10. 加入提交频率限制和队列容量限制。

**验收标准**

- 恶意或错误代码不能访问网络。
- 无限循环会被超时终止。
- 大内存申请会被终止。
- 学生代码不能读取隐藏数据目录外的文件。
- 评测服务崩溃后不会影响数据库一致性。

**安全注意**

Docker 沙箱不是绝对安全边界。正式部署时，不建议 Web 后端直接挂载 Docker socket。更稳妥的做法是把评测 runner 放在独立机器或独立权限域中。

## 七、阶段 5：教师教学闭环

**目标**：让平台帮助教师发现教学问题，而不仅是给学生打分。

**建议工期**：2-3 周。

**主要任务**

1. 教师端班级进度页。
2. 作业通过率和平均分统计。
3. 按案例查看常见错误。
4. 查看学生提交代码、结果和日志。
5. 实验报告提交与教师批注。
6. 人工补评分。
7. 成绩导出为 CSV/XLSX。
8. 支持截止时间、迟交标记、重交次数。

**验收标准**

- 教师能在 1 分钟内看出哪些学生未提交。
- 教师能查看某个实验的失败原因分布。
- 教师能对自动评分不足的部分进行补评分。
- 能导出期末成绩表。

---

## 八、阶段 6：正式部署与运维

**目标**：支撑一个教学班稳定使用。

**建议工期**：2 周。

**主要任务**

1. 部署 HTTPS。
2. 配置生产环境变量和密钥。
3. 数据库每日备份。
4. 提交代码、日志、结果 artifact 备份。
5. 设置服务健康检查。
6. 设置基础监控：CPU、内存、磁盘、评测队列长度。
7. 做 50 人同时提交压测。
8. 编写教师使用手册和故障处理手册。

**验收标准**

- 服务器重启后服务可恢复。
- 数据库可从备份恢复。
- 压测时不会出现大量任务丢失。
- 评测失败有日志可追踪。
- 教师知道如何发布作业、查看提交、导出成绩。

---

## 九、阶段 7：持续扩展与质量提升

**目标**：将平台从课程试点变成可长期复用的实验系统。

**可选增强**

- Redis + BullMQ 异步队列。
- MinIO 对象存储。
- WebSocket 实时评测进度。
- 更多案例自动评测器。
- 代码查重。
- 参数实验记录和对比。
- 自动生成实验报告草稿。
- 错误诊断知识库。
- 课程多学期复制。
- 学生学习画像。
- 排行榜，但建议只用于启发式算法类案例。

**扩展顺序建议**

1. 先扩案例覆盖率。
2. 再扩教师统计能力。
3. 再扩系统并发和对象存储。
4. 最后做排行榜、可视化大屏和智能分析。

## 十二、首批案例接入细节

### case_01：生产分配问题

评测指标：

- 目标值误差。
- 产品变量值误差。
- 资源约束违反量。
- 影子价格误差。
- 是否返回对偶或敏感性分析字段。

建议评分：

| 项 | 分值 |
|----|------|
| 可行性 | 20 |
| 最优性 | 35 |
| 影子价格/对偶理解 | 20 |
| 代码鲁棒性 | 10 |
| 实验报告 | 15 |

### case_04：分配问题

评测指标：

- 每个工人最多/恰好一个任务。
- 每个任务最多/恰好一个工人。
- 非方阵虚拟行列处理。
- 总成本。
- 输出 assignment 格式。

建议评分：

| 项 | 分值 |
|----|------|
| 分配可行性 | 30 |
| 目标值质量 | 35 |
| 非方阵鲁棒性 | 15 |
| 运行效率 | 10 |
| 实验报告 | 10 |

### case_16：模拟退火 TSP

评测指标：

- 路线是否为合法排列。
- 是否回到起点。
- 路线总长度。
- 与基准解的 GAP。
- 运行时间。
- 是否提供收敛曲线。

建议评分：

| 项 | 分值 |
|----|------|
| 路线合法性 | 25 |
| 解质量/GAP | 35 |
| 效率 | 15 |
| 收敛数据和可视化 | 10 |
| 参数分析报告 | 15 |

## 十三、决策建议

短期不要追求“大而全”。最稳的启动方式是：

1. 先完成课程资产标准化。
2. 用 3 个案例验证评测契约。
3. 用最小平台完成一次小班试点。
4. 根据真实课堂反馈决定是否引入队列、对象存储、排行榜和监控。

这样建设出来的平台会更贴合课程，而不是变成一个功能很多但案例难接、教师难用、学生反馈不清楚的通用系统。
